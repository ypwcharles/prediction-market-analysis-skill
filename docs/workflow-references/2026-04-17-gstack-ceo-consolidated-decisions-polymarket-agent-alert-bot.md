## Product Thesis

- This product is not a tracker and not a generic alert bot.
- It is an agent-first Polymarket trading assistant, closer to a "chief of staff" than a dashboard.
- The wedge is `opportunity discovery + judgment`, not raw telemetry.
- V1 is `human-in-the-loop`. It helps decide and monitor, but does not auto-execute trades.
- Telegram is the primary surface.
- A Dockerized alert-bot runtime service is the primary runtime shell.
- Hermes/OpenClaw are optional operator clients, not required runtime owners.

## Locked Scope

Accepted scope additions:

- Stateful thesis memory
- Repricing / reactivation engine
- Telegram feedback and calibration loop
- Automatic position read and overlap-aware recommendations
- Evidence-source and rule-scope explanation cards

V1 scope now means:

- scheduled scanner
- strict/research split
- Telegram alerts
- minimal but action-oriented monitoring
- memory/state layer
- calibration and production gating

Still not in V1:

- autonomous execution
- one-click order placement
- full dashboard product
- broad commercial packaging layer

## Locked Architecture

- Keep `prediction-market-analysis` as a pure judgment engine.
- Build a separate orchestrator for scanning, state, position sync, alerting, and monitoring.
- Use local `SQLite` as the V1 state source of truth.
- Use `Polymarket official API` as the portfolio/position truth source.
- Use a Dockerized alert-bot service as the scheduler/runtime shell.
- Preserve CLI entrypoints for `scan`, `monitor`, `report`, `callback`, and `promote`.
- Add a thin authenticated HTTP control plane for service operations and Telegram webhook ingress.
- Hermes/OpenClaw may call the runtime, but they are not required for the bot to keep running.
- V1 deployment topology is single-instance only:
  - one active runtime instance
  - one writable persistent volume
  - one scheduler owner
  - one Telegram webhook ingress owner
- Use a `dual-speed engine`:
  - `main scan`: every 2 hours
  - `light monitor`: every 15 minutes
  - `heartbeat`: same cadence as main scan

## Locked Opportunity Flow

- Run both `strict mode` and `research mode`.
- Candidate promotion uses a `two-stage flow`:
  - code handles factual and low-dispute filtering
  - LLM handles core upgrade judgment
- The system must not become a pure rule filter.
- The system must also not make the LLM scan the full board raw.

Chosen balance:

- code: activity status, basic liquidity/spread, basic dedupe, field gathering
- LLM: rule audit, evidence synthesis, cluster judgment, edge framing, strict/research routing

## Locked Evidence Policy

Primary evidence contract:

- explicit rule-scope check
- market/executability check
- at least 2 primary evidence sources that corroborate each other
- explicit `why now`
- explicit `kill criteria`

If this contract is not met:

- candidate cannot enter normal `STRICT`
- it can still remain `RESEARCH`

`STRICT-DEGRADED` is allowed, but:

- it must be strongly labeled
- it is an exception path
- it must require stronger edge than normal `STRICT`
- exact uplift can be judgment-based, depending on what evidence is missing

Source policy:

- maintain a layered source registry
- primary evidence comes from rules/platform/original data and approved news sources
- `X` is supplementary and early-signal oriented
- unknown sources cannot independently support `STRICT`

Conflict policy:

- unresolved conflict among primary sources blocks `STRICT`
- conflict can only be cleared with an explicit explanation:
  - higher-authority source
  - later updated source
  - or proof that the sources are answering different questions

## Locked Edge Policy

- `STRICT` baseline promotion currently keys off `theoretical edge > 10c`
- high-priority memos must display both:
  - `theoretical edge`
  - `estimated executable edge`
- they must also display:
  - spread/slippage haircut
  - what basis was used to compute `max entry`

Action rule:

- promotion can be triggered by theoretical edge
- `max entry`, sizing, and "should act now" must be derived from executable edge

## Locked Messaging Policy

- Single Telegram channel
- Strong labels
- Mixed urgency inside one channel

Delivery timing:

- `STRICT`: immediate
- high-value `REPRICE`: immediate
- `RESEARCH`: digest with main scan
- `MONITOR`: only when a concrete action is triggered
- `DEGRADED`: only when it materially affects judgment quality or system state

Message size:

- `STRICT`: long memo
- high-value `REPRICE`: long memo
- `RESEARCH`: short
- `MONITOR`: short
- `DEGRADED`: short

If no strict idea is found:

- still send a short `heartbeat`
- include coverage accounting and system status

## Locked Monitoring Policy

- Monitoring is `action-oriented`, not just price-alerting.
- High-priority memos must generate follow-on monitoring triggers.
- Triggers must be machine-readable, not just free text.
- Trigger schema must be `structured trigger schema + short human explanation`.
- Trigger execution is two-layer:
  - mechanical thresholds: fast monitor can decide directly
  - narrative/evidence/rule drift: must route back through LLM

Trigger lifecycle is stateful:

- armed
- fired
- acknowledged / snoozed
- rearmed / closed

The goal is to avoid repeated noise from the same trigger condition.

## Locked Feedback Policy

Telegram V1 feedback actions:

- `已看`
- `稍后提醒`
- `已下单`
- `不认同`
- `关闭 thesis`

Feedback vs truth:

- Polymarket official API is the position truth source
- Telegram feedback is an intent layer
- `已下单` first becomes pending/user-claimed, then becomes real only after API reconciliation

## Locked Memory Policy

- `thesis cluster` is the memory primary key
- `market_id / token_id` are expression-layer identifiers
- code proposes stable cluster identity
- LLM can merge/split/rename clusters and must record reasons

`关闭 thesis` means:

- close the current thesis version
- do not re-alert on minor price motion or nearby expressions
- only reopen on substantial change, for example:
  - rules changed
  - key evidence flipped
  - price entered a genuinely new tradable zone
  - a much better expression emerged

## Locked Freshness Policy

Every `STRICT` and high-value `REPRICE` must include:

- `evidence_fresh_until`
- `recheck_required_at`
- what happens when freshness expires

TTL policy:

- code provides TTL bounds by archetype / market type
- LLM picks the specific TTL within those bounds and explains why

Expiry policy:

- expired memo becomes `stale / pending-recheck`
- it is no longer a live action recommendation
- fast monitor should not keep issuing actions from stale memo state
- recheck happens later via the normal scan path or a major trigger

## Locked Archive Policy

Use dual-layer storage:

- machine state in `SQLite`
- human-readable high-value artifacts in a private runtime archive layer inside the repo, default gitignored

Archive only high-value artifacts:

- `STRICT` long memos
- high-value `REPRICE` long memos
- short `heartbeat` summaries

Do not archive all research noise as human-readable artifacts.

Archive path strategy:

- repo-internal
- default gitignored
- stable private asset layer
- structured like a long-lived product asset, not a scratch folder

## Locked Citation Policy

There is no local evidence pack in V1.

Instead, high-priority memos must follow strict citation discipline:

- source name
- URL
- fetched time
- which claim this source supports
- if conflict exists, which claim supersedes which claim

This mapping must be written twice:

- in the human-readable memo
- in `SQLite` as structured `claim -> source` mapping

## Locked Calibration Policy

Launch path:

- `shadow`
- `calibration`
- `production`

Production is gated by a formal `calibration report`, not by gut feel.

Calibration must consider both:

- `current quality`
- `ex-post outcomes`

Ex-post review must be multi-timescale:

- short-term: 24h / 72h price and executable changes
- medium-term: entry/add/reduce/exit/kill path
- long-term: final settlement and thesis hindsight

Sample principles:

- do not rely on alert count alone
- require both alert volume and thesis-cluster coverage
- repeated alerts in the same cluster should count partially by default
- only substantial change earns full sample credit again

Exact numeric launch thresholds are not locked yet.

Instead, calibration promotion should be justified through the report itself.

## Compressed Package Resolutions

These packages were intentionally compressed to avoid another 20 micro-questions.
The user chose to accept the default recommendation for all 6 packages.

### Package 1: Source Registry Bootstrap

- Start with a small curated allowlist.
- Store it in repo config, under version control.
- Include:
  - initial primary-news allowlist
  - small authority list for `X`
  - source tier definitions
- Treat expansion of the registry as an explicit future maintenance task, not free-form crawl behavior.

### Package 2: Calibration Report Shape

- Use a fixed report shape.
- Include fixed scorecard dimensions and fixed recommendation states.
- Recommendation states:
  - `not ready`
  - `ready for limited trial`
  - `ready for production`

### Package 3: Runtime Contract

- One slow job and one fast job.
- Shared `SQLite`.
- Explicit env/secrets contract.
- Explicit retry/locking rules.
- Explicit degraded-mode reporting behavior.

### Package 4: Storage and Schema

- First-pass schema should stay narrow.
- Optimize for:
  - operational truth
  - monitoring
  - calibration
  - replayability
- Do not prematurely build broad analytics schema.
- Minimum expected entities:
  - `runs`
  - `alerts`
  - `thesis_clusters`
  - `triggers`
  - `feedback`
  - `positions`
  - `claim_source_mappings`
  - `sources`

### Package 5: Telegram Template Finalization

- Cap V1 at no more than 4 canonical templates.
- Expected set:
  - `STRICT / high-value REPRICE` long memo
  - `RESEARCH` digest
  - `MONITOR` action template
  - `HEARTBEAT / DEGRADED` short system template

### Package 6: Promotion Boundary

- Keep runtime archives private by default.
- Promotion into `docs/market-analysis/` is manual only in V1.
- No automatic promotion rules yet.
