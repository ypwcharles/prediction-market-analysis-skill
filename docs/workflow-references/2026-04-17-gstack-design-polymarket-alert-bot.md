## Problem Statement

Build a self-used, semi-automated Polymarket trading assistant whose core runtime can run as a long-lived Dockerized service and can also be called from agent environments like Hermes or OpenClaw. It scans active and liquid markets on a schedule, identifies high-edge opportunities, gathers external context from news and X-like sources, produces decision-ready trade recommendations, pushes them to Telegram, and also watches existing positions for meaningful changes.

The product is not meant to replace the user's judgment entirely. It is meant to move the heavy lifting earlier in the loop: information gathering, market filtering, rule-scope checking, and base-level edge judgment should happen before the user opens Polymarket.

**Ops Model Update (2026-04-20):** The original 2026-04-17 design assumed Hermes/OpenClaw would remain the scheduler/runtime shell. That assumption is now superseded by the approved Docker-first runtime service model. Hermes/OpenClaw remain valid operator surfaces and integration clients, but they are no longer required to keep the bot running.

## What Makes This Cool

The cool part is not "another dashboard" or "another whale alert bot." The cool part is waking up to 1-3 trades that are already filtered, argued, and sized enough that they are actionable.

This is closer to a Polymarket chief of staff than a tracker. It should tell the user:

- which trades matter
- why they matter
- why the market may be wrong
- what price is still good enough
- what would invalidate the thesis

## Constraints

- V1 must integrate cleanly with agent environments like Hermes or OpenClaw, but they should not be required to own the long-running runtime.
- V1 should push to Telegram. It should not require opening a local dashboard first.
- Discovery is the core wedge. Position monitoring matters, but it is secondary because the user can already do it manually and existing tools partially cover it.
- Recommendation quality and trust matter more than full automation.
- One-click execution is attractive, but it should not define V1.
- The existing `skills/prediction-market-analysis/` asset should be reused, not replaced.

## Definitions

| Term | V1 meaning |
|---|---|
| active/liquid market | a currently open market that passes minimum volume, liquidity, and spread gates defined by the scanner config |
| high-edge opportunity | a candidate whose estimated executable edge survives the configured filter threshold after spread/slippage haircuts |
| decision-ready alert | a Telegram message containing market link, side, thesis summary, maximum entry price, confidence, and kill criteria |
| meaningful position change | a price move, thesis change, or monitoring trigger that crosses a user-configured alert threshold |
| coverage accounting | a run summary that states how many events/contracts were scanned, promoted, rejected, and skipped |
| sparse and useful monitoring | reminders only when a threshold is crossed, not on every ordinary tick |

## Premises

1. V1 should be agent-first, not dashboard-first.
2. The wedge is AI-assisted judgment, not raw telemetry.
3. The product should optimize for trustworthiness of alerts before execution speed.
4. The best architecture is to keep the current prediction-market-analysis skill as the judgment engine and add scanner, orchestration, state, and delivery around it.
5. Telegram is the correct first distribution surface because it matches the user's desired loop.

## Approaches Considered

### Approach A: Agent-First Alert Bot

Keep the existing prediction-market-analysis skill as the analysis core. Add a scheduled scanner/orchestrator that ingests Polymarket markets, filters candidates, enriches survivors with rules and external evidence, calls the judgment engine, and pushes the top opportunities to Telegram. Add lightweight position monitoring and reminder messages.

Pros:

- Best fit for the user's desired first loop
- Smallest product shell that still creates daily value
- Reuses the strongest existing repo asset

Cons:

- Telegram-first interaction limits exploratory drill-down
- Needs careful control of false positives or it becomes noisy fast

### Approach B: Stateful Research Console

Add a persistent state layer and operator console from day one. Track candidate markets, historical fair value estimates, thesis state, user responses to alerts, watchlists, and monitoring history. Telegram becomes one output, not the whole product.

Pros:

- Better long-term product spine
- Stronger memory and less re-analysis
- Easier future open-source or commercial packaging

Cons:

- Higher complexity for V1
- Risks spending too much time on infrastructure before proving daily usefulness

### Approach C: Execution-Centric Trading Terminal

Center the product around one-click buy/sell, position adjustment, and execution workflow. Discovery and monitoring feed directly into a semi-automated trading terminal.

Pros:

- Most complete long-term imagined product
- Tightest decision-to-action loop

Cons:

- Pulls the team into the hardest and riskiest part too early
- Security, failure, and trust concerns all get amplified at once
- Risks weakening the real wedge, which is judgment quality

## Recommended Approach

Choose Approach A.

The product's differentiator is not execution plumbing. It is credible discovery and pre-judgment. The first version should prove that this system can repeatedly surface opportunities the user would not have found, or would have found much later, and package them in a way that is immediately actionable.

The right V1 is:

1. scheduled market scan
2. candidate filtering
3. evidence collection
4. judgment engine call
5. Telegram alert with trade brief
6. minimal position monitoring and reminder logic

## V1 Boundaries

### In V1

- scheduled Polymarket scanner
- filter and rank active/liquid candidates
- evidence collection with pluggable providers
- reuse of `skills/prediction-market-analysis/` as the judgment engine
- Telegram opportunity alerts
- minimal position monitoring for threshold-based reminders
- persisted alert history and last-run state

### Not in V1

- full dashboard or research console
- one-click execution
- autonomous order placement
- heavy portfolio analytics beyond simple overlap/reminder logic
- a broad commercial/open-source packaging layer

## Open Questions

- How much external evidence should be gathered before the system decides a candidate is worth the full analysis pass?
- What exact threshold defines a "high-quality" alert for V1: raw estimated edge, executable net edge, or a composite score?
- How much position monitoring should be built in V1 versus delegated to existing tools?
- Should Telegram alerts include a deep link only, or an execution hint format that a later tool can translate into one-click actions?
- How will X/Twitter-source reliability be bounded so the system does not overfit to noisy social chatter?

## End-to-End Sequence

```text
scheduled run
  -> fetch active/open Polymarket events and markets
  -> apply hard filters (volume, liquidity, spread, market status)
  -> group/cluster related contracts to avoid duplicate same-thesis alerts
  -> enrich survivors with rules, order-book context, and lightweight external evidence
  -> invoke prediction-market-analysis engine on top candidates
  -> score by edge x confidence x liquidity
  -> send top 1-3 Telegram alerts or explicit "no good trades found"
  -> persist run summary, alert history, and monitoring markers
```

## Failure Modes & Guardrails

- If Polymarket market ingestion fails, the run should emit a degraded-status message rather than silent success.
- If news or X providers fail, the scanner should fall back to a degraded evidence mode and either:
  - suppress the candidate, or
  - send a lower-confidence alert explicitly labeled as degraded.
- If the same market was already alerted recently and nothing material changed, dedupe it.
- If overlapping scheduled runs occur, enforce idempotency with a per-run lock.
- If a market resolves, closes, or loses liquidity before alert delivery, suppress it.
- If no candidate clears the threshold, send coverage accounting instead of forcing a pick.
- If position monitoring data is stale or unavailable, monitoring alerts should pause and say so explicitly.

## Provider and Runtime Assumptions

- Polymarket market data comes from Gamma/CLOB-style public endpoints already used in the repo's research workflow.
- News and X-like evidence should be pluggable by provider. V1 should not assume one fragile source.
- A Dockerized alert-bot runtime service provides the primary scheduler/runtime shell. V1 should assume:
  - service-owned scheduled invocation
  - persistent environment variables / secrets
  - lightweight local state storage on a mounted persistent volume
  - optional operator access from Hermes/OpenClaw or future clients
- Persistent storage for V1 can be minimal:
  - last run timestamp
  - recent alert ids for dedupe
  - current open position snapshot
  - candidate/watch markers

## Telegram Alert Example

```text
TRADE CANDIDATE
Market: Fed rate cut by June 29?
Link: https://polymarket.com/event/...
Side: NO
Why now: market repriced on weak headline flow, but rule path and timing evidence still favor miss
Estimated executable edge: 13c
Confidence: medium-high
Max entry: 41c
Kill criteria: confirmed emergency meeting / materially stronger timing evidence / spread widens beyond threshold
Coverage: scanned 86 events, promoted 4, rejected 27, skipped 55
```

## Success Criteria

- The system can run continuously as a single-instance Dockerized service without manual intervention.
- Hermes/OpenClaw can still trigger or inspect the runtime as optional operator clients.
- On a typical run, it evaluates the active/liquid market set and returns either:
  - 1-3 high-quality opportunities, or
  - an explicit "no good trades found" result with coverage accounting.
- Telegram alerts are decision-ready: market link, thesis summary, edge framing, maximum entry price, kill criteria.
- The user feels the alerts are worth reading daily, not just technically correct.
- Position-monitoring messages are sparse and useful, not spammy.
- The system handles degraded-source conditions explicitly instead of silently acting as if evidence was complete.

## Distribution Plan

Initial distribution is private and Docker-first:

- one long-running runtime service
- Telegram as the delivery surface
- Polymarket links for execution
- Hermes/OpenClaw as optional operator clients

CI/CD for V1 can stay light:

- repository-hosted Docker build path
- env-driven deployment configuration
- single-instance runtime operation notes
- simple deployment path for alerting credentials and runtime config
