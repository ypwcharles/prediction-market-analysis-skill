# Eng Review: Polymarket Agent Alert Bot

Generated by `/plan-eng-review` on 2026-04-17

## Verdict

The plan is technically sound if implemented as a `Python-first runtime` inside this repo, with the existing `prediction-market-analysis` skill preserved as a pure judgment engine.

No blocking architecture issue remains after the user accepted the default engineering decisions.

The main engineering constraint is not correctness. It is keeping the runtime boring, narrow, and debuggable while still shipping the full V1 feature set.

## Locked Engineering Decisions

- `runtime/` lives inside this repo as a dedicated product subtree
- `Python-first`
- one runtime package plus one CLI with subcommands
- judgment remains delegated to the skill/prompt asset, not reimplemented as rules
- two scheduled jobs:
  - `scan`
  - `monitor`
- Telegram uses push messages plus inline buttons
- config is versioned in repo
- `pyproject.toml` + `uv`
- full test skeleton in V1
- calibration report writes both `Markdown` and `SQLite` summary
- runtime data lives in a repo-internal gitignored private data layer
- `STRICT-DEGRADED` is a first-class schema/template state
- this round writes the plan only, not code skeleton

### Recommended Package Boundaries

- `scanner/`
  Gamma/CLOB fetch, board discovery, factual prefilter, market normalization
- `sources/`
  source registry, news fetchers, X watchlist fetchers, source-tier handling
- `judgment/`
  context assembly for the skill, invocation adapter, structured parse of result
- `monitor/`
  trigger loading, trigger evaluation, stale/recheck logic, action escalation
- `delivery/`
  Telegram send/edit/button handlers
- `storage/`
  SQLite schema, repositories, locks, migrations
- `templates/`
  memo and digest rendering
- `calibration/`
  report generation and sample accounting
- `archive/`
  high-value artifact persistence into `.runtime-data/archives/`

### Failure Scenarios

| Integration Point | Real failure | Planned handling |
|---|---|---|
| Gamma board fetch | `403`, timeout, partial response | mark run degraded, no fake success |
| CLOB executable check | token lookup/book missing | candidate cannot become normal `STRICT` |
| source fetch | provider rate limit or stale feed | degrade or suppress according to evidence contract |
| skill invocation | malformed or partial LLM output | parse failure -> candidate rejected or degraded, never silent promote |
| Telegram delivery | send failure / callback failure | retry + delivery error state in SQLite |
| Polymarket position sync | stale or missing API result | treat Telegram user action as intent only, not truth |
| overlapping jobs | double-send or write races | SQLite lock + run idempotency key |

### Architecture Outcome

No unresolved architecture issue remains.

The plan should proceed with one code tree, one runtime language, one SQLite, and two jobs.

- Keep `claim -> source` mapping explicit, not hidden in rendered markdown.
- Keep source registry config versioned and human-editable.

## Performance Review

### Main concerns

- repeated board fetch volume
- repeated source fetch volume
- duplicate LLM calls on stale or already-closed thesis clusters
- Telegram callback and monitor races

### Recommendations

- cache run-scoped fetches in-memory inside a single job
- persist enough normalized market metadata to avoid refetching whole board for every monitor cycle
- keep monitor path mechanical by default
- route only narrative triggers back through LLM
- keep archives compact, not raw full-page snapshots

### Performance Outcome

No blocking performance issue exists in the current plan.  
The main protection is architectural: do not let the fast monitor turn into a mini scanner.

## Parallelization Strategy

### Dependency Table

| Step | Modules touched | Depends on |
|------|-----------------|------------|
| Runtime foundation | `runtime/`, `storage/`, `config/`, `.runtime-data/` | — |
| Scanner and sources | `scanner/`, `sources/`, `judgment/` | Runtime foundation |
| Delivery and templates | `delivery/`, `templates/`, `archive/` | Runtime foundation |
| Monitor and callbacks | `monitor/`, `delivery/`, `storage/` | Runtime foundation, Delivery and templates |
| Calibration reporting | `calibration/`, `storage/`, `archive/` | Runtime foundation, Scanner and sources |
| Tests | `tests/` across all modules | all lanes feed fixtures and cases |

### Parallel Lanes

- `Lane A`: Runtime foundation
- `Lane B`: Scanner and sources, then calibration
- `Lane C`: Delivery and templates, then monitor and callbacks
- `Lane D`: Tests trailing all module lanes

### Execution Order

- Start `Lane A` first.
- After `Lane A`, launch `Lane B` and `Lane C` in parallel.
- Merge both.
- Then complete `Lane D`.

### Conflict Flags

- `Lane B` and `Lane C` both eventually touch `storage/` assumptions through shared models. Coordinate schema early.
- `Lane C` monitor work and Telegram callback work both touch delivery/state transitions. Keep them sequential inside the same lane.
